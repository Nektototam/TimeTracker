# Правила разработки и лучшие практики

## 1) Качество кода

- Использовать TypeScript строго.
- Избегать бизнес-логики в UI-компонентах.
- Все side-effects — в сервисах/хелперах.
- Во фронтенде запрещены самодельные UI-компоненты. Использовать только Shadcn UI.

## 2) TDD / тестирование

- Новая функциональность: сначала тест, потом код.
- Минимум: unit-тесты для сервисов, интеграционные для API.
- Smoke-тесты для критических маршрутов.

## 3) Код-ревью

- Без одобрения минимум одного ревьюера — merge запрещен.
- PR должен содержать: описание, скриншоты (если UI), тесты.

## 4) Git и ветки

- main — только стабильный код.
- feature/* — новые фичи.
- fix/* — исправления.
- chore/* — техдолг.

## 5) Definition of Done

Задача считается готовой, если:
- Есть тесты (или явное обоснование их отсутствия).
- Линт и тесты проходят.
- Нет регресса в ключевых сценариях.
- Документация обновлена.

## 6) Безопасность

- Пароли только через bcrypt.
- Секреты только в env.
- Rate-limit на auth.

## 7) Открытый репозиторий

- Никаких ключей/токенов в git.
- Использовать только .env.example с пустыми значениями.
- Проверять PR на утечки секретов.

## 8) Документация

- Архитектурные решения фиксируются в docs/.
- Обновлять схему БД при каждом изменении.
